# Target data model

## Conventions

- All application primary keys use UUIDs generated by PostgreSQL/Prisma.
- Public references use separate unique numbers or slugs; internal IDs are not customer-facing identifiers.
- Financial fields use `Decimal @db.Decimal(19, 4)` and explicit currency codes.
- Mutable business entities use `createdAt`, `updatedAt`, and optional `deletedAt` where archival is required.
- Status changes that matter operationally have append-only history records rather than relying only on a current enum.
- PostgreSQL `citext` or normalized companion columns are considered for case-insensitive email/SKU uniqueness during migration design.

## Identity and RBAC

- `User`: email, password hash, verification and account status.
- `Account`, `Session`, `VerificationToken`: Auth.js persistence.
- `Role`, `Permission`, `UserRole`, `RolePermission`: many-to-many RBAC. `RolePermission.effect` is `ALLOW` or `DENY`; deny wins.
- `CustomerProfile`, `CompanyProfile`, `Address`: retail/business identity and South African address fields.
- `AuditLog`: immutable actor/action/target record with JSON summaries.

## Catalogue and procurement

- `Category`: self-referencing parent, slug, ordering, SEO and active state.
- `Brand`: slug, logo, content and active state.
- `Product`: SKU, slug, descriptions, prices, tax state, publication flags, warranty, SEO and soft deletion.
- `ProductImage`, `ProductDocument`, `ProductSpecification`: ordered media and metadata.
- `ProductVariant`: variant SKU, price overrides, barcode and purchasable state.
- `ProductAttribute`, `ProductAttributeValue`: reusable option definitions; a variant-to-value join records combinations.
- `Supplier`, `SupplierProduct`: supplier details and product-specific cost, SKU, stock, MOQ, lead time, preference and last update.

Price snapshots are copied to cart/order/quotation lines. Historical documents must never depend on the current product price or description.

## Inventory

- `Inventory`: one row per variant (or base product when no variants), with on-hand, reserved, incoming and reorder quantities.
- `InventoryMovement`: immutable ledger for receipts, reservations, releases, sales, returns, corrections and damage.

Availability is `onHand - reserved`; it is derived, not independently editable. Stock reservation locks the inventory row in a database transaction, verifies availability, increments reserved, and appends a movement. Cancellation/payment expiry releases the reservation idempotently.

## Cart, promotions and wishlist

- `Cart`: optional user association plus a cryptographically random anonymous token and expiry.
- `CartItem`: product/variant, quantity and optional pricing snapshot for display only.
- `Wishlist`, `WishlistItem`: one customer list with unique items.
- `Coupon`, `CouponRedemption`: eligibility, scope, dates, usage limits and immutable redemption linkage.

The server reloads products, variants, stock, VAT settings and coupons before calculating checkout totals.

## Orders, payments and fulfilment

- `Order`: unique order number, customer/contact snapshot, subtotal, discount, delivery, VAT, grand total, statuses and notes.
- `OrderItem`: immutable product, SKU, variant, quantity, unit/cost/discount/VAT/line totals and description snapshots.
- `OrderAddress`: immutable billing/delivery snapshots.
- `OrderStatusHistory`: append-only transition record.
- `Payment`: provider, external reference, amount, status, idempotency key and normalized metadata.
- `PaymentProof`: private object reference, review result, reviewer and timestamps.
- `Shipment`: carrier, tracking, status and delivery timestamps.
- `Invoice`: numbered immutable document metadata and storage reference.

Valid state transitions live in the order/payment domains. Webhooks never set arbitrary statuses directly.

## Quotations

- `QuotationRequest`, `QuotationRequestItem`, `QuotationRequestAttachment`: customer need before pricing.
- `Quotation`, `QuotationItem`: versioned price snapshot, terms, expiry, totals and numbered PDF.
- `QuotationStatusHistory`: append-only lifecycle.

Conversion to an order occurs once in a transaction, copies immutable snapshots, reserves inventory, links the records, and returns the existing order on an idempotent retry.

## Engagement, content and operations

- `Review`: rating, moderation state, response and verified-purchase flag.
- `HomepageBanner`, `Page`: managed storefront content and SEO fields.
- `SiteSetting`: typed/configured settings with sensitive payment or banking values access-restricted and audited.
- `Notification`: durable in-app/email notification intent and delivery status.
- `RecentlyViewedProduct`: bounded customer or anonymous history.
- `NewsletterSubscription`: normalized email, consent source/status and timestamps.

## Required indexes

At minimum:

- Unique normalized user email; product SKU/slug; variant SKU; category/brand slug; business document numbers.
- Product publication/category/brand/created date and featured/special/new combinations.
- Inventory low-stock query fields and movement `(inventoryId, createdAt)`.
- Orders `(userId, createdAt)`, `(status, createdAt)`, `(paymentStatus, createdAt)` and number.
- Payments `(provider, externalReference)` and idempotency key.
- Quotations/request status and customer/date combinations.
- Audit target/date and actor/date combinations.
- Coupon code and active date range.

Indexes will be validated with real query plans rather than added indiscriminately.

## Transaction boundaries

| Workflow | Atomic operations |
| --- | --- |
| Checkout | Validate authoritative prices/stock/coupon; create order snapshots; reserve stock; redeem coupon; initialize payment record |
| Payment confirmation | Lock payment/order; verify amount/provider event; transition status; append histories; enqueue notifications |
| Cancellation | Validate transition; release remaining reservations; cancel payment where possible; append ledger/history |
| Refund | Validate refundable amount; create refund record; update aggregate state; optionally return stock; audit |
| Quotation conversion | Lock accepted quotation; create order snapshots; reserve stock; link once; append histories |
| Manual inventory adjustment | Lock inventory; adjust on-hand; append movement with reason and actor |

External calls are never held inside database transactions. An outbox/idempotency pattern coordinates post-commit email, PDF, and payment operations.
